Agents, Evals, and Orchestration

The difference between a script that calls an LLM and an agent that reasons about what to do next. And how to know if any of it actually works.

Overview

The word "agent" is used carelessly in AI right now. Half the industry uses it to mean "chatbot with a system prompt." That is not what we mean. An agent is a system that can observe its environment, reason about what to do, take action, and observe the result -- in a loop, autonomously, until it accomplishes a goal or decides to stop.

In Module 2, you gave Claude tools. That was a single round: user asks a question, Claude calls a tool, you return the result, Claude answers. An agent takes that pattern and runs it in a loop. It decides which tools to use, in what order, based on intermediate results it has not seen yet. It can change course. It can recover from errors. It can decide when it is done.

This module also covers something most agent tutorials skip entirely: evaluation. If you cannot measure whether your agent is actually working -- not just generating plausible-sounding text, but producing correct, useful results -- you are building on sand. Evals are not glamorous. They are essential.

What Makes Something an Agent

Scripts vs agents

A script with an LLM call has a fixed execution path. Step 1: take user input. Step 2: call Claude. Step 3: display response. The flow is predetermined. The LLM generates text, but does not decide what happens next.

An agent has a dynamic execution path. It reads the situation, decides what to do, does it, evaluates the result, and decides what to do next. The key difference: the LLM is making decisions about control flow, not just generating content.

Consider the difference:

  • Script: "Summarize this document" -- one API call, done.
  • Agent: "Research the impact of remote work on productivity and write a report" -- the agent decides which sources to search, evaluates whether it has enough information, identifies gaps, searches again, synthesizes findings, drafts sections, and decides when the report is complete.

The agent loop

Every agent, from the simplest to the most sophisticated, follows the same core loop:

  1. Observe: Read the current state. What does the agent know? What has happened so far? What tools are available?
  2. Think: Given the observations, what should happen next? This is where Claude reasons about strategy.
  3. Act: Execute the chosen action -- call a tool, write output, request information.
  4. Observe again: What was the result of the action? Did it succeed? Did it change what the agent knows?
  5. Repeat or stop: Is the goal accomplished? If not, go back to step 2.

In code, this is a while loop. Each iteration sends the accumulated context to Claude, including previous actions and their results. Claude decides the next action. Your code executes it. The loop continues until Claude signals completion or you hit a safety limit (max iterations, max tokens, max time).

Critical safety note: Always implement a maximum iteration count. An agent without a stop condition can run forever, burning tokens and potentially taking unintended actions. Start with a low limit (10 iterations) and increase only when you understand what each iteration is doing.

Building a Simple Agent

The minimal architecture

A working agent needs surprisingly little code. The core components:

  • A system prompt that defines the agent's role, goals, and available tools
  • A message history that accumulates observations and actions
  • A loop that sends the history to Claude and processes the response
  • Tool execution logic from Module 2
  • A termination condition (Claude says it is done, or max iterations reached)
messages = [{"role": "user", "content": task}]

for i in range(max_iterations):
    response = client.messages.create(
        model="claude-sonnet-4-20250514",
        system=agent_prompt,
        tools=available_tools,
        messages=messages,
        max_tokens=4096
    )

    # Add Claude's response to history
    messages.append({"role": "assistant", "content": response.content})

    # Check if Claude used a tool
    tool_calls = [b for b in response.content if b.type == "tool_use"]

    if not tool_calls:
        break  # Claude responded with text only — task complete

    # Execute tools, add results to history
    tool_results = []
    for call in tool_calls:
        result = execute_tool(call.name, call.input)
        tool_results.append({
            "type": "tool_result",
            "tool_use_id": call.id,
            "content": str(result)
        })

    messages.append({"role": "user", "content": tool_results})

That is a working agent loop. Everything else -- error handling, logging, state management, persistence -- is refinement of this core pattern.

Evaluations: Does It Actually Work?

Why evals matter more than architecture

Here is an uncomfortable truth about AI development: it is easy to build something that looks like it works. Claude generates fluent, confident text. Your agent runs through its loop and produces output that reads well. But is the output correct? Is it better than what a simpler approach would produce? Is it consistent? Without evaluation, you are guessing.

Evals are how you turn guessing into knowing. They are the tests for AI systems -- not unit tests that check if a function returns the right value, but systematic measurements of whether your system produces good outputs across a range of inputs.

Types of evals

  • Exact match: Did the agent return the correct answer? Works for factual questions, math, code that must pass tests.
  • LLM-as-judge: Use a separate Claude call to evaluate the output. "Given this question and this answer, rate the quality from 1-5 and explain your reasoning." This is more flexible but introduces its own biases.
  • Human evaluation: Have a person review outputs. The gold standard, but expensive and slow. Use it to calibrate your automated evals.
  • Task completion: Did the agent achieve the goal? For an agent that researches and writes reports, the eval might check: did it cite real sources? Did it cover the required topics? Is the report the requested length?

Building an eval suite

Start simple. Create 10-20 test cases: inputs with known good outputs. Run your system on all of them. Measure the results. This is your baseline. Every change you make -- to prompts, tools, architecture -- should be tested against this suite. If performance drops, you know before your users do.

The eval mindset: If you cannot eval it, you cannot improve it. The builders who get hired are not the ones who build the most complex systems -- they are the ones who can demonstrate, with evidence, that their systems work. Evals are that evidence.

Orchestration Patterns

Once you can build one agent, the question becomes: how do you compose multiple AI calls into larger systems? These are the foundational patterns.

Sequential (pipeline)

Output of step 1 feeds into step 2, which feeds into step 3. Example: research agent gathers information, writing agent drafts the report, editing agent refines the prose. Simple, predictable, easy to debug.

Parallel (fan-out/fan-in)

Multiple independent tasks run simultaneously, results are combined. Example: search three different sources at the same time, then synthesize the results. Faster, but requires logic to merge potentially conflicting information.

Router

A classifier decides which specialized agent handles the request. Example: a customer service system routes billing questions to one agent, technical questions to another, complaints to a third. Each agent has different tools, prompts, and expertise.

Supervisor

A higher-level agent manages and coordinates lower-level agents. It assigns tasks, reviews results, and decides whether to accept the work or send it back for revision. This is the most powerful pattern and the most complex. Use it only when simpler patterns are genuinely insufficient.

When agents are overkill

Most of the time. A single, well-prompted API call with good tools handles the vast majority of use cases. If you find yourself building a multi-agent system to answer questions that a single Claude call could handle, stop and simplify. Agents add latency, cost, and complexity. They are justified when the task genuinely requires multi-step reasoning with intermediate results that cannot be predicted in advance. For everything else, a script with an LLM call is not just simpler -- it is better.

Exercise: Build a Research Agent with Evals

Build an agent that can research a topic, summarize its findings, and write a short report. Then build evals that measure whether the reports are actually good.

The agent should:

  • Accept a research topic as input
  • Have access to at least one information-gathering tool (web search API, Wikipedia API, or a local knowledge base you create)
  • Make multiple tool calls to gather information from different angles
  • Produce a structured report with an introduction, key findings, and conclusion
  • Decide on its own when it has gathered enough information

The evals should measure:

  • Does the report address the requested topic? (relevance)
  • Does it contain specific facts rather than vague generalities? (depth)
  • Is it well-structured with clear sections? (format)
  • Does the agent complete within a reasonable number of tool calls? (efficiency)

Create at least 5 test topics and run your agent on all of them. Record the scores. This is your baseline -- when you improve the agent's prompts or tools, re-run the evals and compare.
