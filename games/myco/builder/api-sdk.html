API and SDK Fundamentals

The interface between your code and Claude's intelligence. This is where you stop being a user and start being a builder.

Overview

Every tool you have used so far -- Claude.ai, Claude Code, any chatbot interface -- is a wrapper around the same thing: the Anthropic API. When you type a message in Claude.ai, your browser sends an HTTP request to Anthropic's servers. A model processes it. A response comes back. That is the entire magic trick.

In this module, you will learn to make that request yourself. No wrapper. No UI. Just your code talking to Claude directly. This is the single most important skill in the Builder path, because everything else -- tool use, agents, MCP, production systems -- is built on top of API calls.

You completed the Foundation. You know how to think with Claude, how to craft prompts, and when not to use AI at all. Now you learn the plumbing.

The Anthropic API

What it is

An API (Application Programming Interface) is a contract: you send a request in a specific format, you get a response in a specific format. The Anthropic API lets you send messages to Claude models and receive responses programmatically -- from scripts, applications, servers, or anything that can make an HTTP request.

The API is stateless. Claude does not remember your previous conversations through the API. Every request must include the full conversation history you want Claude to see. This is different from Claude.ai, where the interface manages conversation history for you. Understanding this distinction early will save you hours of confusion later.

Getting an API key

Go to console.anthropic.com. Create an account. Generate an API key. This key is a secret -- treat it like a password. Never commit it to a git repository, never paste it in a public forum, never put it in client-side code that ships to browsers.

Store it in an environment variable:

export ANTHROPIC_API_KEY="sk-ant-..."

Your code reads from the environment. Your key stays out of your source files. This is not optional hygiene -- it is a fundamental security practice you must internalize now, before you have a key with real spend on it.

Cost reality check: Anthropic provides a small amount of free credits to start. After that, API access costs money -- you pay per token. For learning and building small projects, the cost is minimal (a few dollars a month). But you should always know what you are spending. We cover pricing in detail below.

The Messages API

Anatomy of a request

The Messages API is the core endpoint. Every conversation with Claude flows through it. A request has three essential parts:

  • Model: Which Claude model to use (e.g., claude-sonnet-4-20250514). Different models have different capabilities, speeds, and costs. Sonnet is the workhorse. Opus is the most capable. Haiku is the fastest and cheapest.
  • Messages: An array of conversation turns, alternating between user and assistant roles. This is how you give Claude context and history.
  • System prompt: Optional but powerful. Sets Claude's behavior, persona, and constraints for the entire conversation. This is where the prompt craft you learned in Foundation becomes infrastructure.
{
  "model": "claude-sonnet-4-20250514",
  "max_tokens": 1024,
  "system": "You are a helpful coding assistant. Be concise.",
  "messages": [
    {"role": "user", "content": "What is a REST API?"}
  ]
}

Multi-turn conversations

Because the API is stateless, multi-turn conversations require you to send the full history each time. Round one, you send one user message. Round two, you send the user message, Claude's response, and the new user message. The message array grows with every turn.

This is why context windows matter. Claude has a maximum number of tokens it can process in a single request. The longer your conversation history, the more tokens you consume per request. You will learn to manage this -- sometimes by summarizing earlier turns, sometimes by being selective about what history to include, sometimes by starting fresh.

Streaming

By default, the API waits until Claude finishes generating its entire response before sending anything back. For long responses, this means your user stares at nothing for seconds. Streaming sends the response token by token as it is generated -- the same effect you see in Claude.ai where text appears progressively.

Streaming is not just a cosmetic nicety. For any user-facing application, it is essential. A three-second wait with progressive text feels fast. A three-second wait with nothing, then a wall of text, feels broken. The difference between these two experiences is about ten lines of code.

Choosing Your Stack

Python SDK

Install it: pip install anthropic. Python is the default choice for AI work because the ecosystem is deep, the documentation assumes it, and the majority of tutorials and examples use it. If you have no strong preference, start here.

import anthropic

client = anthropic.Anthropic()

message = client.messages.create(
    model="claude-sonnet-4-20250514",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "Explain APIs to someone who is smart but not technical."}
    ]
)
print(message.content[0].text)

That is a complete, working program. Five lines of meaningful code. The SDK reads your ANTHROPIC_API_KEY from the environment automatically.

TypeScript SDK

Install it: npm install @anthropic-ai/sdk. Choose TypeScript if you are building web applications, already know JavaScript, or want type safety enforced at compile time. The SDK mirrors the Python one closely -- concepts transfer directly between them.

The SDK handles the tedious parts

You could make raw HTTP requests to the API with curl or fetch. The SDKs exist so you do not have to. They handle authentication headers, request formatting, response parsing, automatic retries on transient errors, and streaming. Use them. Writing raw HTTP calls to the Anthropic API is like writing your own TCP stack -- technically possible, practically pointless.

Tokens, Pricing, and Rate Limits

What tokens are

Claude does not process words -- it processes tokens. A token is roughly three-quarters of a word in English. "Hello, how are you today?" is about 7 tokens. A 1,000-word essay is roughly 1,300 tokens. You pay for both input tokens (what you send to Claude) and output tokens (what Claude generates), with output tokens costing more.

Pricing intuition

Exact prices change, so check the Anthropic pricing page for current numbers. But develop intuition: a short question-and-answer exchange costs a fraction of a cent. Sending a 50-page document for analysis costs noticeably more. Generating a 3,000-word report costs more still. The most expensive pattern is sending large context windows repeatedly -- like re-sending an entire book with each follow-up question instead of extracting the relevant section.

Rate limits

The API limits how many requests you can make per minute and how many tokens you can process per minute. New accounts have lower limits that increase over time and with higher usage tiers. When you hit a rate limit, the API returns a 429 status code. Your code must handle this gracefully -- not crash, not silently fail, but wait and retry.

Error Handling and Retry Strategies

APIs fail. Networks time out. Servers return errors. Rate limits kick in. The difference between a toy project and a real tool is how it handles failure.

  • 429 (Rate Limited): Wait and retry. Use exponential backoff -- wait 1 second, then 2, then 4, then 8. The SDK handles basic retries automatically, but understand the pattern so you can tune it.
  • 500/503 (Server Error): Retry with backoff. These are transient -- the server had a bad moment.
  • 400 (Bad Request): Do not retry. Your request is malformed -- fix your code.
  • 401 (Unauthorized): Your API key is wrong or missing. Do not retry. Fix your configuration.
The builder's mindset: Every error is information. Log it, understand it, handle it. The fastest way to signal that you know what you are doing is to build things that fail gracefully instead of crashing with a stack trace. This applies far beyond API calls.

Exercise: Build a Streaming CLI Tool

Build a command-line tool that takes a question as input and streams Claude's answer to the terminal in real time.

Requirements:

  • Accept a question as a command-line argument or via interactive prompt
  • Use the Anthropic SDK (Python or TypeScript -- your choice)
  • Stream the response token by token to the terminal
  • Include a system prompt that gives Claude a specific persona or constraint
  • Handle errors gracefully -- if the API key is missing, say so clearly instead of crashing
  • Print token usage (input and output tokens) after the response completes

Stretch goals:

  • Support multi-turn conversation (keep asking follow-up questions in a loop)
  • Let the user choose the model via a command-line flag
  • Save conversation history to a JSON file

This is not a large project. It should be 50-100 lines of code. The point is to internalize the API contract -- request in, response out, errors handled -- by building something real with your hands.
