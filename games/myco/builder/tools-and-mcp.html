Tool Use, Structured Outputs, and MCP

Claude can do more than generate text. It can call functions, return structured data, and connect to the systems you build around it.

Overview

In Module 1, you learned the API contract: send messages, get text back. That is powerful, but it has a hard ceiling. A model that can only produce text cannot check your database, cannot call your APIs, cannot take action in the world. It is an oracle locked in a room -- brilliant, but unable to reach anything outside its own training data.

Tool use breaks that ceiling. You define functions that Claude can call. Claude decides when to call them, with what arguments, based on the conversation. Your code executes the function and sends the result back. Claude incorporates the result into its response. Suddenly the oracle can reach into the real world.

This module covers three connected ideas: tool use (function calling), structured outputs (getting reliable data formats instead of prose), and MCP (the protocol that standardizes how Claude connects to external systems). Together, they transform Claude from a text generator into a component you can wire into real software.

Tool Use: Function Calling

How it works

The mechanism is straightforward. When you make an API call, you include a tools parameter -- a list of functions Claude is allowed to call. Each tool has a name, a description, and a schema that defines its parameters. Claude reads the conversation, decides whether any tool would help, and if so, responds with a tool_use block instead of (or alongside) regular text.

tools = [{
    "name": "get_weather",
    "description": "Get the current weather for a given city.",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {
                "type": "string",
                "description": "The city name, e.g. 'San Francisco'"
            }
        },
        "required": ["city"]
    }
}]

When a user asks "What's the weather in Tokyo?", Claude does not guess or hallucinate a weather report. It returns a structured request: call get_weather with {"city": "Tokyo"}. Your code catches that request, actually checks the weather (via an API, a database, whatever), and sends the result back. Claude then writes its response using real, current data.

The tool use loop

This is the pattern you need to internalize:

  1. You send a message to Claude with tools defined
  2. Claude responds -- either with text, or with a tool_use block (or both)
  3. If Claude wants to use a tool, your code executes the function
  4. You send the result back as a tool_result message
  5. Claude generates its final response incorporating the tool result

This loop can repeat. Claude might call one tool, get a result, decide it needs another tool, call that one, and then synthesize everything into a response. Your code orchestrates this loop. Claude decides the logic.

Designing good tool schemas

The quality of your tool definitions directly determines how well Claude uses them. This is a form of prompt engineering applied to function signatures.

  • Names should be verb-noun: get_weather, search_database, send_email. Not weather or data_thing.
  • Descriptions should explain when to use the tool: "Get the current weather for a city. Use this when the user asks about weather conditions, temperature, or forecasts." Claude uses these descriptions to decide which tool to call.
  • Parameters should have their own descriptions: Do not just say "city": {"type": "string"}. Say "city": {"type": "string", "description": "City name, e.g. 'San Francisco, CA'"}. Examples in descriptions dramatically improve accuracy.
  • Mark required fields as required: If a parameter is optional, Claude should know. If it is mandatory, enforce it in the schema.
Opinionated take: Tool design is where most builders fail. They define vague tools with unclear boundaries and then blame the model when it uses them wrong. A well-described tool with clear parameters is worth more than a clever prompting trick. Spend time here.

Structured Outputs

The problem with prose

Ask Claude to "list the top 5 cities by population" and you will get a nicely formatted paragraph or bulleted list. That is great for a human reader. It is terrible for your code. Parsing natural language to extract structured data is fragile, error-prone, and unnecessary -- because Claude can give you structured data directly.

Getting JSON from Claude

You can instruct Claude to return JSON in its response via the system prompt ("Always respond in JSON format"), but this is unreliable. The robust approach is to use tool use as a structured output mechanism: define a tool whose schema matches the data structure you want, and Claude will "call" it with perfectly formatted data.

tools = [{
    "name": "format_cities",
    "description": "Format a list of cities with their populations.",
    "input_schema": {
        "type": "object",
        "properties": {
            "cities": {
                "type": "array",
                "items": {
                    "type": "object",
                    "properties": {
                        "name": {"type": "string"},
                        "country": {"type": "string"},
                        "population": {"type": "integer"}
                    },
                    "required": ["name", "country", "population"]
                }
            }
        },
        "required": ["cities"]
    }
}]

Claude returns data that matches your schema. Your code parses it with confidence. No regex. No hoping the format is consistent. This pattern is essential for any application where Claude's output feeds into downstream code.

MCP: The Model Context Protocol

What MCP is

Tool use as described above works, but it requires you to define every tool inline in your API calls and write all the execution logic yourself. MCP (Model Context Protocol) is an open standard that changes this. It defines a universal way for AI models to connect to external data sources and tools -- a shared protocol, like how HTTP standardized web communication.

With MCP, tools live in servers -- small programs that expose capabilities through the protocol. A Slack MCP server exposes "send_message" and "read_channel" tools. A database MCP server exposes "query" and "insert" tools. A filesystem MCP server exposes "read_file" and "list_directory" tools. Your AI application connects to these servers and gains their capabilities automatically.

Why MCP matters

Before MCP, every AI application had to build its own integrations from scratch. Want Claude to access your database? Write the tool definition, write the execution code, handle errors, manage connections. Want it to access Slack too? Do it all again for a different API. Every integration was custom plumbing.

MCP standardizes this. Build an MCP server once, and any MCP-compatible client can use it. The ecosystem grows combinatorially: every new server benefits every client, and every new client benefits from every server. This is the same dynamic that made the web powerful -- the protocol creates a platform.

Building your first MCP server

An MCP server is simpler than you might expect. Using the official Python or TypeScript SDK, you define tools as decorated functions. The SDK handles the protocol negotiation, message formatting, and communication. A minimal MCP server that exposes a single tool is roughly 30 lines of code.

The important conceptual shift: you are not just calling Claude anymore. You are building infrastructure that Claude (and other AI systems) can use. You are on the other side of the API, providing capabilities rather than consuming them.

Where the industry is headed: MCP adoption is accelerating. Claude Code, Cursor, Windsurf, and an expanding list of AI tools support MCP natively. The builders who understand this protocol now -- who can build servers, who can think in terms of composable tool ecosystems -- are positioning themselves for where AI development is going, not where it has been.

Putting It Together

The progression across this module maps to real capability levels:

  • Tool use means Claude can take action beyond generating text
  • Structured outputs mean Claude's responses can be consumed by code, not just humans
  • MCP means these capabilities are standardized, reusable, and composable

Most "AI applications" in the wild today are simple wrappers: take user input, add a system prompt, call the API, display the response. With tool use and MCP, you can build applications where Claude reasons about which actions to take, executes those actions through well-defined interfaces, and synthesizes the results. This is a qualitative leap in what you can build.

Exercise: Build a Tool-Using Application

Build an application that gives Claude access to at least one external tool and uses it to answer questions it could not answer with training data alone.

Choose one (or combine them):

  • A weather tool that calls a free weather API (OpenWeatherMap, wttr.in) and lets Claude answer "What's the weather in Berlin?"
  • A calculator tool that performs precise math, so Claude does not have to approximate
  • A database lookup tool that queries a SQLite database of your choosing (movies, books, recipes -- anything)
  • A file reader tool that lets Claude read and summarize files from your local filesystem

Requirements:

  • Define at least one tool with a clear schema (name, description, typed parameters)
  • Implement the tool use loop: send request, detect tool call, execute, return result
  • Handle the case where Claude does not need to use the tool (not every question requires one)
  • Print or log the tool calls Claude makes, so you can see the reasoning

Stretch goal: Implement this as an MCP server instead. Expose your tool through the MCP protocol so that any MCP client (Claude Code, for example) can use it.
