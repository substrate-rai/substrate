Prompt Craft as a Real Skill

"Prompt engineering" sounds like a secret art. It's not. It's clear thinking expressed in writing — and it's a skill you can develop systematically.

Overview

The internet is full of "prompt engineering" content that treats prompting as a collection of magic incantations. Use this secret phrase. Add this special word. Copy this template. Most of it is nonsense.

The truth is simpler and more useful: a good prompt is clear communication. The same skills that make you good at writing an email to a colleague — clarity, context, specificity, knowing your audience — make you good at prompting Claude. This module teaches you the principles behind effective prompts so you can craft your own instead of copying someone else's.

Core Content

Clear Thinking, Not Magic Words

Here's the uncomfortable truth about prompt engineering: the prompt isn't the hard part. The thinking is.

When someone gets a bad response from Claude, they almost always blame the prompt. "I didn't phrase it right." But usually the real problem is upstream — they hadn't clearly defined what they wanted, who it was for, or what constraints mattered. The prompt was vague because their thinking was vague.

This means the best way to get better at prompting isn't to memorize tricks. It's to get better at answering four questions before you type anything:

  1. What specifically do I want Claude to produce?
  2. What does Claude need to know to produce it well?
  3. What constraints should it operate within?
  4. How will I know if the result is good?

If you can answer those four questions, the prompt practically writes itself.

The Anatomy of a Good Prompt

Every effective prompt has four components. You don't always need all four, but knowing what they are lets you diagnose what's missing when results aren't right:

  • Context: What does Claude need to know about the situation? Background information, who the audience is, what's already been tried, what domain you're in. Claude can't read your mind — it can only work with what you give it.
  • Task: What specifically should Claude do? "Write," "analyze," "compare," "explain," "debug," "brainstorm." Use a clear verb. One task per prompt, wherever possible.
  • Constraints: What are the boundaries? Length, tone, format, things to avoid, things to include, level of detail. Constraints aren't limitations — they're focusing tools.
  • Format: What should the output look like? Bullet points, numbered list, table, prose paragraph, JSON, code with comments. Being explicit about format saves you from having to reformat later.

Here's the difference these components make, using a real example:

Bad PromptTell me about marketing.
Okay PromptExplain three digital marketing strategies for a small business with a limited budget.
Good PromptI run a one-person freelance web design business. My monthly marketing budget is $200. Most of my clients come from referrals, but referrals are slowing down. Suggest 3 specific marketing strategies I could start this month. For each one: - What it is and why it works for solo freelancers - Estimated time investment per week - How to measure if it's working after 30 days Keep the tone practical and direct. Skip anything that requires a team or significant upfront investment.

The bad prompt gives Claude nothing to work with. The okay prompt is better but still generic. The good prompt gives context (solo freelancer, $200 budget, referrals drying up), a clear task (3 strategies), constraints (startable this month, no team required), and format (structured with specific sub-points). Claude can actually produce something useful from this.

System Prompts vs. User Prompts

If you're using Claude through the API or any tool that lets you set a system prompt, it helps to understand the distinction. If you're just using claude.ai for conversations, you can skip this section for now — but you'll need it eventually.

A system prompt sets the overall context, role, and rules for the entire conversation. Think of it as the job description you give Claude before the work starts. "You are a technical editor. You focus on clarity and conciseness. You flag jargon and suggest simpler alternatives."

A user prompt is the specific request within that context. "Edit this paragraph for clarity."

The system prompt stays constant across the conversation. The user prompt changes with each turn. Putting persistent instructions (tone, role, format preferences) in the system prompt keeps your user prompts cleaner and more focused. It also tends to be more effective — Claude treats system prompt instructions as higher-priority context.

Few-Shot Examples: Show, Don't Just Tell

Sometimes the fastest way to get what you want isn't to describe it — it's to show it. Few-shot prompting means giving Claude one or more examples of the input-output pattern you want, then asking it to follow the pattern with new input.

This is especially powerful when:

  • You want a very specific format or style that's hard to describe in words
  • You're doing a classification or categorization task
  • You need consistency across many similar items
  • The task involves domain-specific conventions Claude might not default to

One or two good examples often work better than a paragraph of instructions. Three examples is usually enough for Claude to lock onto the pattern. If you need more than five, the pattern might be too complex for a single prompt — consider decomposing the task.

Chain of Thought: When to Ask Claude to Think Step by Step

Asking Claude to "think step by step" or "reason through this before answering" isn't a trick — it reflects something real about how language models work. When Claude generates intermediate reasoning steps before reaching a conclusion, it tends to make fewer errors on complex tasks.

Use chain of thought when:

  • Math or logic problems — Claude is significantly more accurate when it shows its work
  • Multi-step analysis — when the answer depends on several factors that interact
  • Ambiguous situations — when there are multiple valid interpretations and you want Claude to consider them explicitly
  • Debugging — asking Claude to trace through code line by line rather than jump to a conclusion

Don't bother with chain of thought for simple tasks. Asking Claude to "think step by step" about what day comes after Tuesday just adds noise. Match the technique to the difficulty of the problem.

Temperature and Parameters: What They Actually Control

If you use the API or any tool that exposes model parameters, you'll encounter temperature. Here's what it actually does, stripped of the mystery:

Temperature controls how much randomness is in Claude's output. At low temperature (close to 0), Claude picks the most statistically likely next word at each step. The output is more predictable, consistent, and focused. At higher temperature (closer to 1), Claude is more willing to pick less-likely words, making the output more varied, creative, and occasionally surprising.

Practical guidance:

  • Low temperature (0-0.3): Use for factual tasks, code generation, data extraction, anything where consistency and accuracy matter more than creativity.
  • Medium temperature (0.4-0.7): Good default for most tasks. Balanced between consistency and variety.
  • High temperature (0.8-1.0): Use for creative writing, brainstorming, generating diverse ideas. Accept that some outputs will be weird — that's the point.

Other parameters like max tokens (controls response length) and top-p (another way to control randomness) exist, but temperature is the one that matters most for day-to-day use. If you're using claude.ai, you don't control these directly — Anthropic sets sensible defaults.

Key Takeaway

Prompt craft isn't a separate skill from thinking clearly. It's the same skill, applied to a new medium. Every time you write a better prompt, you're also getting better at articulating what you want — a skill that pays off everywhere, not just with AI.

Exercise

The Three-Version Prompt

Pick a task you genuinely need done. Something from your work or a personal project — not a made-up example. Then write three versions of the same prompt:

  1. Version 1 — Bad: Write the laziest version. One vague sentence, no context, no constraints. The kind of prompt you'd fire off without thinking.
  2. Version 2 — Okay: Add specificity. Include what you want and basic constraints, but don't provide much context about your situation.
  3. Version 3 — Great: Apply the full anatomy. Context about your situation, clear task, specific constraints, explicit format. Include a few-shot example if relevant.

Run all three with Claude and compare the outputs side by side.

Write down:

  • How much better was the great prompt's output vs. the bad prompt's?
  • Which of the four components (context, task, constraints, format) made the biggest difference?
  • What did you learn about your own thinking process by forcing yourself to be explicit?

Save your three prompts and the results. You'll want to reference them as you develop your own prompting style.
